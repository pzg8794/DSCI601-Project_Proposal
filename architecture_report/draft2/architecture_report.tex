\begin{abstract}
This draft presents the repository as a push-validated CRISP-DM review system
rather than as a small generic machine-learning pipeline. The architecture is
organized around four interacting concerns: repository events that introduce new
work, control mechanisms that orchestrate and validate that work, CRISP-DM
phases realized as executable stages, and an artifact layer that produces the
review evidence used in merge decisions. This framing is important because the
project is evaluated not only on whether it runs, but on whether new work can be
checked reproducibly as it is pushed. The resulting architecture emphasizes
methodological traceability, deterministic execution, reviewer-visible
artifacts, and a merge-check path that links local validation to continuous
integration.
\end{abstract}

\section{Architecture Overview}

This architecture writeup should be read as the operational companion to the
project's earlier approach diagram. The earlier approach artifact explains the
project's conceptual logic and research direction. The architecture presented
here explains how that logic is realized as a repository that checks new work as
it is pushed, runs the same review path locally and in CI, and exposes
sufficient evidence for another party to inspect or challenge the result.

That distinction matters. A project can have a sound conceptual method and still
be architecturally weak if its software does not preserve stage boundaries,
artifact lineage, or reproducible review. The goal of this draft is therefore
not to redescribe the project in broad terms, but to show how the repository
embeds CRISP-DM into a concrete review workflow.

The repository should be described as \textit{CRISP-DM-aligned}. CRISP-DM gives
the project its methodological spine, while the software decomposes that spine
into executable review stages that a reviewer can run independently or
end-to-end. In practical terms, business understanding appears in the form of
review goals, fairness thresholds, and merge criteria; data understanding is
realized through collection and exploration; data preparation is split into
cleaning and preparation; modeling is realized by the mining stage; evaluation
remains explicit; and deployment is represented by reviewer-facing outputs,
manifest generation, and merge-readiness checks.

The architecture therefore has a dual obligation. It must preserve the research
logic of the project, and it must preserve the software logic needed for review.
A weak draft can satisfy one while missing the other. This draft is designed to
keep both visible at once.

\section{Design Principles}

The architecture is governed by four design principles.

\textbf{Methodological traceability.} The repository should remain legible in
CRISP-DM terms even when one conceptual phase is implemented through more than
one executable stage.

\textbf{Deterministic reviewability.} A reviewer should be able to rerun a
compact configuration and obtain the same structure of outputs without depending
on hidden state.

\textbf{Artifact-centered evidence.} The value of the run lies not only in a
final number or figure, but in the chain of intermediate artifacts that justify
a merge decision.

\textbf{Separation of concerns.} Repository events, orchestration, validation,
stage execution, and evidence publication should be visible as separate
responsibilities rather than hidden inside a monolithic script.

\subsection{Event Layer}
The left side of the architecture represents the point at which work enters the
system. In this project, that entry point is not just a human pressing run on a
single script. New work arrives through repository pushes, pull requests,
configuration changes, and explicit developer or reviewer commands such as
\texttt{run-all} and \texttt{merge-check}. The event layer therefore answers a
basic architectural question: when does the repository become active as a review
system?

The answer is that a change is treated as a request for validation. In a local
workflow that request is handled by the command-line interface. In a shared
workflow the same request is mirrored by CI. This is why the architecture must
be described in push-validated terms rather than as a static pipeline drawing.

\subsection{Control and Review Layer}
The control layer translates repository events into a structured execution plan.
It includes configuration loading, command routing, stage ordering,
reproducibility controls, test invocation, and merge policy. This layer is the
software realization of project governance: it defines what should run, under
what settings, and what must be true before work is considered review-ready.

Two paths coexist in this layer.

\textbf{Execution path.} The pipeline runs its ordered stages and records their
outputs in a manifest.

\textbf{Validation path.} Tests and merge checks confirm that the execution path
produced structurally correct outputs and that the repository remains healthy.

That separation is essential. A project can execute without being reviewable,
but it cannot be reviewable unless validation is architected explicitly.

realizes the conceptual phases through reviewable software boundaries, as shown

This explicit mapping is one of the main strengths of the architecture. CRISP-DM
is not invoked as a label only. It is translated into code paths that can be run
and inspected. Some conceptual phases expand into multiple stages because
reviewability benefits from finer separation. Data understanding, for example,
remains split between collection and exploration so a reviewer can inspect both
what enters the system and what was learned about it before modeling begins.

Likewise, deployment is interpreted in the context of this deliverable. Here it
does not primarily mean production serving. It means publishing reviewer-facing
evidence: processed outputs, figures, tables, manifests, and merge-ready proof
that the pipeline executed correctly.

\subsection{Artifact and Provenance Layer}
The bottom of the architecture collects the products that matter to a reviewer:
configuration snapshots, raw and prepared data, predictions, metrics, summary
tables, figures, and the final manifest. These outputs are not accidental side
effects. They are the objects through which the repository becomes auditable.

The architecture therefore treats artifacts as first-class entities. Each phase
produces evidence, and that evidence can be followed forward into the final
merge decision. This gives the repository a research-quality property that the
previous draft failed to show clearly: the architecture is not only about
execution order, but about provenance.

\section{Component View}

The main architectural components can be summarized as follows.

\textbf{Repository interface.} This includes the repository events, the CLI,
and the reviewer-facing commands that trigger execution.

\textbf{Configuration subsystem.} This defines deterministic run parameters such
as seed, split ratio, threshold values, missingness controls, and output paths.

\textbf{Pipeline stages.} These implement the numbered transformations that
realize the CRISP-DM lifecycle in executable form.

\textbf{Validation subsystem.} This includes unit tests, smoke checks,
merge-check execution, and CI mirroring.

\textbf{Artifact and provenance subsystem.} This writes stage outputs and the
final manifest so every run leaves an inspectable record.

The architectural claim is not simply that these components exist. It is that
they are visible as separate responsibilities. That visibility is what makes the
repository understandable under review.

\section{Why this Draft Improves the Previous Version}

The earlier draft was weak because it drew a compact stage sketch where a true
system architecture was needed. That choice made the work look smaller and less
coherent than it actually is. More importantly, it blurred the relationship
between CRISP-DM, repository validation, and merge-readiness.

This draft corrects that in three ways.

\textbf{First, it restores system scale.} The diagram now shows repository
triggers, orchestration, CRISP-DM execution, evidence generation, and merge
decision flow in one architecture.

\textbf{Second, it reconnects the architecture to the project's broader
approach narrative.} The architecture is presented as the operational
realization of the earlier conceptual diagram rather than as an unrelated
artifact.

\textbf{Third, it foregrounds reviewability.} The repository is described as a
system for checking work as it is pushed, not merely as a sequence of data
processing steps.

\section{Conclusion}

A strong architecture writeup should show how methodological intent, executable
stages, artifact generation, and review decisions fit together. That is the
central purpose of this draft.

The repository should therefore be understood as a push-validated CRISP-DM
system whose main promise is reproducible review. New work enters through
repository events, passes through orchestration and validation, is processed by
explicit CRISP-DM phases realized as software stages, and produces evidence that
supports a merge decision. That framing better represents the work of the
project and is more appropriate for a research-quality architecture deliverable.

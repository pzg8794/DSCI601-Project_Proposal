\section{Architecture Overview}

This report presents the project architecture as a reproducible,
CRISP-DM-aligned system built on top of the existing
\texttt{quantum\_project} infrastructure. The purpose of the report is to
identify the major layers of the system, document the components that implement
each stage, and provide enough workflow detail for another party to recreate the
software, execute the pipeline, and replicate the reported results.

The inherited quantum workflow provides the main execution structure used here:
shared-drive storage, notebook-based analysis, evaluator-owned summaries,
runner-based experiment execution, and state-analysis postprocessing. The
current repository extends that structure by adding a review-oriented control
layer and a fairness--performance synthesis layer that compares fairness
criteria against inherited performance metrics.

The architecture is organized around five concerns: CRISP-DM context, control
and validation, execution stages, fairness--performance synthesis, and artifacts
and replication outputs.

\section{Recreating the System and Replicating Results}

The architecture supports three entry paths so that another party can recreate
the system and reproduce the reported behavior.

The notebook path is the fastest route to replication because the shared
notebooks already mount Drive, install dependencies, load saved evaluator or
framework state, and write results back to the shared infrastructure. The local
path remains necessary for repository review, testing, and debugging. The GCP
path extends the same workflow to longer or larger runs without changing the
underlying component structure.

These entry paths lead into the same inherited execution chain. They do not
create separate systems. For that reason, the report documents setup, execution,
interpretation, and sharing as part of the architecture rather than as
supplementary instructions.

\section{System Architecture}

Figure~\ref{fig:architecture} summarizes the full workflow. At the highest
level, Business Understanding and Data Understanding remain contextual layers
rather than CI stages. The control layer receives repository events or user
entry paths, applies configuration and validation rules, and drives the
execution stages. The execution layer performs collection, cleaning,
preparation, exploration, mining, and evaluation. The fairness--performance
layer then combines inherited performance summaries with repository-side
fairness criteria. Finally, the artifacts layer preserves the evidence produced
by each stage.

The figure is placed on its own landscape page at the end of the report so that
all layers, arrows, and stage labels remain legible.

\section{Detailed Description of Each Stage}

\subsection{Business Understanding}
Business Understanding defines the research objective, the fairness criteria,
the acceptance rules, and the replication target. It establishes what the
system is expected to optimize and what evidence must be produced before results
are considered review-ready.

\subsection{Data Understanding}
Data Understanding defines the structure of the shared-drive data lake, the
semantics of the saved state, and the assumptions that notebooks and analysis
scripts make about evaluator outputs. The repository depends on a real upstream
storage model rather than an isolated local file layout.

\subsection{Data Collection Stage}
\textbf{Components.} The collection stage is built from
\texttt{LocalBackupManager}, \texttt{GoogleDriveBackupManager}, and the
registry-based latest-state lookup path listed in Table~\ref{tab:stage_objects}.

\textbf{Linkage.} \texttt{GoogleDriveBackupManager} resolves the shared-drive
source and identifies the relevant storage root. \texttt{LocalBackupManager}
provides a local-facing wrapper around that storage path. The registry logic
then indexes the available saved states and selects the latest compatible
artifact for the requested component or run. Together, these parts convert the
shared-drive namespace into a concrete set of executable inputs for the
downstream pipeline.

\textbf{Output.} The stage produces source references and retrieved framework or
model states that can be passed into evaluator-based validation.

\subsection{Data Cleaning Stage}
\textbf{Components.} The cleaning stage is centered on
\texttt{MultiRunEvaluator}, supported by \texttt{\_infer\_models\_from\_state()},
\texttt{\_\_eq\_\_()}, and \texttt{ensure\_summary\_contract()}.

\textbf{Linkage.} The retrieved state from the collection stage is first
inspected by \texttt{\_infer\_models\_from\_state()} so that the system can
recover the model identity and expected evaluator structure. \texttt{\_\_eq\_\_()}
and related evaluator-side checks verify structural compatibility. When
evaluator summaries are incomplete or not normalized,
\texttt{ensure\_summary\_contract()} repairs the summary shape so that later
analysis stages consume a stable payload. Together, these components transform
raw retrieved state into evaluator-valid state.

\textbf{Output.} The output is a cleaned and contract-consistent evaluator state
that downstream analysis can trust.

\subsection{Data Preparation and Exploration Stage}
\textbf{Components.} This stage is built from \texttt{state\_analysis.py} and
the evaluator-owned \texttt{scenarios\_stats} and
\texttt{scenarios\_results} payloads referenced in
Table~\ref{tab:stage_objects}.

\textbf{Linkage.} \texttt{MultiRunEvaluator} provides the scenario summaries
and normalized result structures. \texttt{state\_analysis.py} then consumes
those evaluator outputs and restructures them into flattened tables,
scenario-oriented summaries, and analysis-ready views. The stage is therefore a
composition of evaluator-generated summaries plus state-analysis postprocessing
rather than an independent replacement for evaluator logic.

\textbf{Output.} The output is a prepared set of scenario summaries, tables,
and exploratory artifacts that support mining, notebooks, and reviewer
inspection.

\subsection{Modeling and Mining Stage}
\textbf{Components.} The mining stage is implemented through
\texttt{AllocatorRunner}, \texttt{QuantumExperimentRunner}, and the allocator
objects responsible for route qubit allocation.

\textbf{Linkage.} The allocator objects define the decision logic used within
each run. \texttt{QuantumExperimentRunner} executes the underlying experiment
path and produces run-level outputs and traces. \texttt{AllocatorRunner}
orchestrates the higher-level execution across allocators, scenarios, and
configurations. Together, these components produce the substantive outputs that
the evaluator later summarizes.

\textbf{Output.} The stage produces model outputs, traces, and experiment
artifacts suitable for scenario-level comparison.

\subsection{Evaluation Stage}
\textbf{Components.} The evaluation stage is owned by
\texttt{MultiRunEvaluator}, particularly through
\texttt{calculate\_scenario\_winner()},
\texttt{calculate\_scenario\_performance()}, and
\texttt{calculate\_scenarios\_performance()}.

\textbf{Linkage.} The evaluator consumes the runner outputs, organizes them by
scenario, and computes the summary statistics required for comparison: winner
counts, average gap, average efficiency, oracle reward, and related
scenario-level measures. This stage is deliberately evaluator-owned so that
judgment logic remains close to the object that also maintains the state
contract.

\textbf{Output.} The output is a reviewer-visible performance summary that can
be consumed directly or passed into the fairness--performance layer.

\subsection{Fairness--Performance Synthesis Layer}
\textbf{Components.} This new layer combines a repository-side
fairness--performance component with allocator context, runner outputs,
evaluator metrics, and visualizer or robustness-plotting consumers such as
\texttt{QuantumEvaluatorVisualizer}.

\textbf{Linkage.} The runner chain and allocator context define how
performance was produced. The evaluator computes the inherited performance
metrics. The fairness--performance component then places those metrics beside
the fairness criteria defined at the repository level and produces tradeoff
views that neither the raw runner nor the raw evaluator produces on its own.
The visualizer path then consumes those combined outputs to generate figures and
comparative summaries.

\textbf{Output.} The output is a set of tradeoff tables, comparative plots, and
review-facing fairness-versus-performance summaries.

\subsection{Review Delivery and Replication Outputs}
\textbf{Components.} The final stage uses notebooks, \texttt{visualizer.py},
shared-drive exports, and \texttt{pipeline\_manifest.json}.

\textbf{Linkage.} The prior stages generate the saved state, metrics,
fairness summaries, and plots. The notebook and visualizer path turns those
into human-readable artifacts. The manifest records provenance for the run.
Shared-drive export keeps the outputs accessible to collaborators and later
reanalysis.

\textbf{Output.} The output is a set of figures, tables, rerunnable notebook
artifacts, and provenance records that support review and replication.

\section{User Workflow: Setup, Run, Interpret, and Share}

\subsection{Setup and Installation}
A user can interact with the system through three supported paths.

\textbf{Notebook path.} The user opens the shared-drive notebook, mounts Google
Drive, installs dependencies in the notebook environment, and runs the relevant
cells. This is the fastest path for reproducing reported results.

\textbf{Local path.} The user clones the repository, creates a virtual
environment, installs dependencies from \texttt{requirements.txt}, and runs a
smoke or validation script.

\textbf{GCP path.} The user provisions or enters a VM, runs the startup script,
and then uses the batch or dynamic runner scripts for longer experiments.

\subsection{How the User Runs the System}
Once setup is complete, the user chooses a stage or full workflow. In the local
path, this usually means running the quick validation or full review path. In
the notebook path, it means configuring an experiment in the notebook and
executing the evaluator-backed workflow. In the GCP path, it means invoking the
startup and batch scripts and letting the run complete remotely.

These paths are different entry points into the same inherited execution
structure. They do not define different systems.

\subsection{How the User Interprets Results}
The user interprets results through the same artifact conventions used by the
upstream project. Framework-state and model-state outputs are written under the
shared data lake, often in day-partitioned directories. Notebooks and visualizer
logic then consume those results to generate plots, scenario comparisons, and
summary tables.

\subsection{How the User Shares Results}
Because the data lake is shared-drive based, completed runs can be made visible
to collaborators without inventing a second export pipeline. Results written to
\texttt{quantum\_data\_lake/}, including framework state, model state, tables,
and figures, are already placed in a common location that supports team
inspection and notebook-based reanalysis.

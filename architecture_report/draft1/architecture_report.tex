\section{Architecture Overview}
This document describes the architecture of the DSCI~601 code-review repository,
structured as a \textbf{push-triggered CRISP+ML pipeline}: every \texttt{git push}
to a monitored branch runs all seven CRISP-DM stages and the unit-test suite via
CI, enforcing that each stage produces valid artifacts before a pull request is
allowed to merge.

The pipeline is grounded in the real quantum networking research toolchain rather
than synthetic data. It inherits the \texttt{GoogleDriveBackupManager} for data
collection from the quantum data lake, the \texttt{MultiRunEvaluator} object for
cleaning and evaluation, \texttt{state\_analysis} utilities for preparation and
mining, and \texttt{visualizer.py} for postprocessing.

\subsection{Stage~1 --- Data Collection}
The \texttt{DataCollectionStage} fetches real experiment artifacts from the quantum data lake hosted on the team's Google Shared Drive:
\url{https://drive.google.com/drive/u/0/folders/12mZyGQHD__G54g2yCYIyWfXydhO3DxIt}.
It uses \texttt{GoogleDriveBackupManager.build\_registry()} and \texttt{get\_latest\_state(...)} to resolve and load the latest saved state.

\subsection{Stage~2 --- Data Cleaning}
The \texttt{DataCleaningStage} mirrors \texttt{MultiRunEvaluator} validation behavior,
including \texttt{\_infer\_models\_from\_state()} and the evaluator equality contract used to reject invalid saved states.

\subsection{Stage~3 and Stage~4}
\texttt{state\_analysis} is highlighted in two roles: deterministic train/test preparation and exploration summaries such as experiment conditions and scenario statistics.

\subsection{Stage~5 through Stage~7}
\texttt{QuantumExperimentRunner} performs the lightweight mining step, \texttt{MultiRunEvaluator} computes efficiency/fairness evaluation, and \texttt{visualizer.py} handles final postprocessing outputs.

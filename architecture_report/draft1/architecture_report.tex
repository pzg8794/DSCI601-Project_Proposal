\section{Architecture Overview}
%% ─────────────────────────────────────────────────────────────

This writeup describes the implementation of the code-review repository for the
DSCI~601 assignment. The repository is organized as a reproducible
\textit{CRISP+ML} pipeline that mirrors a real machine-learning workflow while
remaining fast enough for reviewers to install, run, test, and inspect locally.
The design goal is a reviewer-safe system with clear stage boundaries,
deterministic behavior, and enough implementation detail that another party can
recreate the workflow and reproduce the same outputs from a single configuration
file.

The system is structured around seven explicit lifecycle stages that directly
instantiate the six canonical phases of the Cross-Industry Standard Process for
Data Mining (CRISP-DM)~\cite{wirth2000crisp}: data collection, data cleaning,
data preparation, data exploration and visualization, data mining, evaluation,
and results postprocessing and visualization.
Table~\ref{tab:crispmap} makes the mapping explicit.

\begin{table}[h]
\centering
\caption{Mapping of CRISP-DM phases to CRISP+ML pipeline stages.}
\label{tab:crispmap}
\small
\begin{tabular}{@{}lp{5.2cm}@{}}
\toprule
\textbf{CRISP-DM Phase} & \textbf{Realized by} \\
\midrule
Business Understanding  & \texttt{configs/fast\_review.json} + external proposal artifacts \\
Data Understanding      & Stages 1 (Collection), 2 (Cleaning), 4 (Exploration) \\
Data Preparation        & Stage 3 (Preparation) \\
Modeling                & Stage 5 (Data Mining) \\
Evaluation              & Stage 6 (Evaluation) \\
Deployment              & Stage 7 (Postprocessing) + CI + publish workflow \\
\bottomrule
\end{tabular}
\end{table}

The architecture is therefore not only a package layout---it is a data pipeline:
each CRISP+ML stage consumes a concrete artifact, produces the next artifact,
and is covered by a corresponding unit-test boundary. This is the key design
choice that lets reviewers inspect the work as a reproducible system rather
than as disconnected scripts.

%% ─────────────────────────────────────────────────────────────
\section{System Architecture}
%% ─────────────────────────────────────────────────────────────

The architecture is organized as a five-layer system. The top layer maps each
stage to its corresponding CRISP-DM phase. The second layer is the cross-cutting
infrastructure---configuration, CLI, and validation---that controls, exposes,
and tests the pipeline. The third layer is the CRISP+ML pipeline itself, with
sequential data flow from collection through postprocessing. The fourth layer
shows the quantum-project object each stage inherits, grounding each CRISP-DM
step in a real research artifact. The bottom layer is the quantum data lake and
pipeline manifest that record data provenance for the entire run.

Because the figure now carries substantial stage, inheritance, and provenance
detail, it is placed on its own dedicated landscape page at the end of this
document so it can remain large and easy to read without competing with the
main writeup text.

\subsection{Configuration Layer}
The configuration layer defines the parameters controlling each run. In the
implementation repository, a typed configuration object is loaded from a single
JSON file. The configuration records the random seed, record counts,
train/test split, invalid-record frequency, missingness frequency, bias level,
fairness threshold, and output directory.

This layer separates experiment settings from stage logic. A reviewer can
change the run size, seed, or output location without modifying source code.
This design also supports reproducibility: the same configuration and seed
produce the same stage artifacts.

\subsection{Stage Layer}
Each lifecycle step is implemented as its own concrete class under a shared
\texttt{PipelineStage} interface. This keeps the implementation object-oriented
and gives every phase a narrow, testable responsibility. The stage layer is the
most important part of the architecture because it directly maps the
methodology to code:

\begin{itemize}
    \item \textbf{Data Collection:} \texttt{DataCollectionStage} retrieves
    real experiment artifacts from the quantum data lake, a Google Shared Drive
    hosted at \url{https://drive.google.com/drive/u/0/folders/12mZyGQHD__G54g2yCYIyWfXydhO3DxIt}.
    By inheritance from the quantum project, it uses
    \texttt{LocalBackupManager} (which extends \texttt{GoogleDriveBackupManager})
    to call \texttt{build\_registry()} and \texttt{get\_latest\_state()} and
    resolve the latest saved \texttt{MultiRunEvaluator} or model state for a
    given component and date partition
    (\texttt{quantum\_data\_lake/\{component\}/day\_YYYYMMDD/}).
    \item \textbf{Data Cleaning:} \texttt{DataCleaningStage} mirrors the
    validation contract of \texttt{MultiRunEvaluator}: it calls
    \texttt{\_infer\_models\_from\_state()} to reconstruct model identity from
    a saved state file, applies the evaluator equality contract (\texttt{\_\_eq\_\_})
    to reject structurally invalid or incompatible states, and writes only
    schema-valid records. This stage is separate so preprocessing is not
    hidden inside modeling code.
    \item \textbf{Data Preparation:} \texttt{DataPreparationStage} mirrors
    the \texttt{state\_analysis} decomposition pattern: it extracts scenario
    statistics (\texttt{scenarios\_stats}), partitions records by scenario
    key, and writes deterministic train/test artifacts under a fixed seed,
    exactly as the quantum project separates stochastic from baseline
    scenarios before model evaluation.
    \item \textbf{Data Exploration / Visualization:}
    \texttt{DataExplorationStage} mirrors \texttt{state\_analysis} exploration
    utilities (\texttt{display\_experiment\_conditions()},
    \texttt{print\_summary()}) to write lightweight summaries of dataset size,
    group balance, scenario coverage, and feature completeness before modeling.
    \item \textbf{Data Mining:} \texttt{DataMiningStage} inherits the
    \texttt{QuantumExperimentRunner} pattern: it invokes \texttt{run\_experiment()}
    to produce reward traces and prediction artifacts, keeping the stage
    intentionally lightweight so the review workflow remains fast.
    \item \textbf{Evaluation:} \texttt{EvaluationStage} mirrors
    \texttt{MultiRunEvaluator} evaluation methods, including
    \texttt{calculate\_scenario\_winner()} for efficiency ranking and
    fairness-oriented group-aware summary values needed by the project.
    \item \textbf{Results Postprocessing / Visualization:}
    \texttt{ResultsPostprocessingStage} inherits the \texttt{visualizer.py}
    pattern from the quantum project to convert evaluation outputs into
    final reviewer-facing CSV tables, SVG figures, and a Markdown summary report.
\end{itemize}

The implementation treats these classes as pipeline stages, not helper
functions. Each stage has an input contract, an output contract, and tests that
verify the expected artifact exists and has the expected structure.

\subsection{Quantum Object Inheritance}
Each CRISP+ML stage is grounded in a real object from the quantum networking
research codebase~\cite{quantumhub}, inheriting its interface by design rather
than reimplementing equivalent logic from scratch.
\texttt{DataCollectionStage} inherits \texttt{LocalBackupManager} /
\texttt{GoogleDriveBackupManager} (registry-based data lake access).
\texttt{DataCleaningStage} and \texttt{EvaluationStage} both inherit
\texttt{MultiRunEvaluator} (state validation contract and scenario winner
calculation, respectively).
\texttt{DataPreparationStage} and \texttt{DataExplorationStage} both inherit
\texttt{state\_analysis} utilities (scenario decomposition and experiment
condition reporting).
\texttt{DataMiningStage} inherits \texttt{QuantumExperimentRunner} (experiment
execution and reward trace production).
\texttt{ResultsPostprocessingStage} inherits \texttt{visualizer.py} (final
chart and CSV export conventions).
This inheritance layer is shown in the full architecture figure and documented
in full in the component reference (\texttt{components.md}).

\subsection{Context and Orchestration Layer}
The orchestration layer connects stages without tightly coupling their
implementations. A shared pipeline context carries metadata, artifact paths, and
stage results across the run. The \texttt{ReviewPipeline} object builds the
ordered stage list and executes it either end-to-end or until a named stopping
point.

After execution, the orchestrator writes a manifest file containing the active
configuration, stage outputs, and stage summaries. This manifest is the
reproducibility record for a run: it identifies what was executed, where
artifacts were written, and what each stage reported.

The same orchestrator supports both normal execution and review execution.
Individual commands can stop at a stage for debugging, while the full
\texttt{run-all} path executes the entire chain. This design makes the repo
usable by both a developer checking one phase and a reviewer checking the whole
submission.

\subsection{Interface Layer}
The user-facing interface is a command-line module with one command for each
required stage plus two orchestration commands: \texttt{run-all} and
\texttt{merge-check}. The command-line interface is part of the architecture
because it makes the repository reviewable as software rather than as a single
notebook. Reviewers can execute a single stage, rerun the full workflow, or
perform the same merge validation path used by the repository.

The \texttt{merge-check} command is the local quality gate. It runs the quick
pipeline and the unit tests so submitted changes can be checked before merging.
The remote CI workflow mirrors this path, making local and pull-request
validation consistent.

\section{Framework and Component Choices}
%% ─────────────────────────────────────────────────────────────

The implementation intentionally uses a minimal technical stack. The pipeline
is written with standard-library Python and a small package structure, with
\texttt{requirements.txt} as the reviewer-facing installation path. This choice
reduces environmental friction and avoids requiring a heavy machine-learning
stack for a code review assignment.

JSON is used for configuration, records, predictions, evaluation summaries, and
manifests because it is human-readable, easy to validate, and supported without
third-party libraries. CSV is used for summary tables because it is familiar to
reviewers and easy to inspect. SVG is used for lightweight charts because it
provides visual output without requiring plotting dependencies.

These choices prioritize reproducibility, transparency, and short setup time
over raw performance. That tradeoff is appropriate for this deliverable: the
goal is not to train a large model, but to demonstrate that the project has a
clean architecture, separable code paths, tests, and a reliable validation
workflow.

\section{Data Storage and Reproducibility}
%% ─────────────────────────────────────────────────────────────

All artifacts are written beneath a stable output root defined by the active
configuration. Within that root, each stage writes to its own numbered
directory. This output design makes the workflow easy to inspect: every phase
leaves visible artifacts, intermediate outputs can be reused, and reviewers can
verify stage behavior without rerunning the entire system. The major artifacts
are raw records, clean records, prepared train/test splits, exploration
summaries, predictions, evaluation metrics, final tables, SVG figures, and the
pipeline manifest.

The output layout answers three practical questions: what configuration
produced this run, what each stage wrote, and what final metrics or figures
were generated. The final manifest connects those answers into a single
provenance record. Because the repository uses deterministic configuration and
fixed seeds, another party can reproduce the same artifacts by running the same
commands.

The expected recreation path is straightforward: create a fresh Python
environment, install from \texttt{requirements.txt}, run the full pipeline with
\texttt{configs/fast\_review.json}, inspect the generated results directory,
and run \texttt{merge-check}. This reproduces the reviewer-safe workflow, its
stage artifacts, and its validation checks.

%% ─────────────────────────────────────────────────────────────
\section{Conclusion}
%% ─────────────────────────────────────────────────────────────

The architecture emphasizes simplicity, determinism, and inspectability. It
uses a recognized CRISP+ML lifecycle as the backbone, object-oriented stage
classes as the implementation structure, and explicit command-line entry points
as the reviewer interface. This makes the system appropriate for the DSCI 601
code review assignment while also providing a reusable foundation for larger
fairness-aware data science workflows.

The explicit CRISP-DM mapping in Table~\ref{tab:crispmap} and the five-layer
architecture figure together ensure that the methodology is not implicit but
structurally visible at every level of the implementation. This traceability---from a recognized data-science process model down to
individually runnable CLI commands, numbered artifact directories, and a
deterministic pipeline manifest---is the property that makes this system
suitable for both the DSCI~601 code-review assignment and for extension into
larger fairness-aware sequential decision-making workflows.

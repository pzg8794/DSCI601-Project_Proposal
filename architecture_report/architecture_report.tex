\section{Architecture Overview}

This work describes the project's architecture as a reproducible, CRISP-DM-aligned research system built on top of our existing \texttt{quantum\_project} infrastructure~\cite{quantumproject}. The goal is not only to list software components. The goal is to show why the same architecture can study two target applications---clinical decision routing and quantum-network routing---without treating either domain as a superficial example.

The inherited quantum workflow reuses the same shared-drive storage, notebook path, evaluator logic, runner path, and state-analysis conventions that already produce reproducible artifacts in the upstream project. The current implementation extends that inherited workflow into a multi-environment fairness framework. Legacy quantum runs remain compatibility-first: the old runner, evaluator, allocator, and state paths still execute the quantum experiments, while fairness analysis attaches as a governed sidecar. In parallel, the clinical path is now embedded as a real experiment path with a clinical environment, runner, evaluator, allocator, model registry, and EQUITAS mediation seam. Both domains now feed the same canonical fairness, reporting, and validation layers.

The central methodological claim is that both target applications are instances of the same scientific problem: sequential allocation under incomplete and unequally reliable context. In clinical settings, the routed resource is diagnostic attention, test distribution, retesting capacity, escalation, or model-assisted triage across sites and cohorts. In volatile conditions such as a pandemic, threat, demand, evidence quality, and resource availability can shift rapidly, making the clinical environment non-stationary and chaotic in ways that are structurally comparable to quantum routing under noisy links and uncertain entanglement generation. In quantum networks, the routed resource is entanglement-generation effort, route capacity, or qubit allocation. In both settings, the system observes partial context, chooses an action, receives feedback only for the chosen action, and risks compounding harm when the least reliable context belongs to the groups or flows that are already hardest to serve. That is why the architecture must connect problem formulation, data characterization, execution, fairness mediation, and report validation rather than stopping at a runner diagram.

The architecture is organized around six interacting concerns: research problem formulation, data and context characterization, execution, environment adaptation and sidecar normalization, fairness--performance reporting, and artifacts/validation. This organization follows the spirit of CRISP-DM but translates it into research terms: the early phases define the harm mechanism and evidence requirements, the middle phases execute comparable experiments, and the later phases preserve enough provenance to defend every table and figure.

Operationally, the architecture also depends on four repository-side mechanisms that make the workflow reviewable as software: a configuration-controlled execution path, a local validation and merge-check path, a manifest-based provenance record, and path-scoped CI smoke gates. These mechanisms allow another party to reproduce a run from a fixed configuration, execute the same validation logic locally, and inspect the exact artifacts written by the workflow.

\section{How the Architecture Interacts with the Target Applications}

The architecture interacts with the target applications through an environment contract rather than through a single hard-coded simulator. Each application supplies four things: an environment or saved state source, a model or policy family, an allocator/action mechanism, and an evaluator-facing outcome record. The framework then normalizes those outcomes into a shared fairness/reporting surface.

In the quantum application, the environment is the inherited quantum-routing stack. Quantum experiments continue to use the original runner and evaluator behavior so that legacy quantum testbeds and saved-state paths from \quantumPaper remain valid. The fairness layer does not rewrite those experiments. Instead, a smoke wrapper executes the old path, maps legacy testbed identifiers to source labels, extracts evaluator states, and writes fairness sidecar artifacts under \texttt{datalake/fairness/quantum/...}. This design answers the main compatibility question: the fairness framework can inspect old quantum results without disrupting old quantum runs.

In the clinical application, the environment is an embedded synthetic clinical fixture. The clinical path runs through \texttt{ClinicalExperimentEvaluator}, \texttt{ClinicalExperimentRunner}, \texttt{ClinicalEnvironment}, clinical resource allocation, clinical model objects, and the EQUITAS mediator. The clinical models include \texttt{Oracle}, \texttt{ClinicalBaseline}, \texttt{FairAllocatorV1}, and \texttt{ClinicalBanditUCB}. Each model produces decisions and outcomes that are converted into the same canonical fairness state format used by the quantum sidecar. This design answers the plug-in question: a non-quantum environment can be inserted into the same architecture and evaluated with the same reporting and validation tools.

The interaction pattern is therefore:

\begin{center}
\small
\begin{tabular}{c}
Application environment or saved state \\
$\downarrow$ \\
Runner / model / allocator decision path \\
$\downarrow$ \\
Evaluator or adapter normalization \\
$\downarrow$ \\
Canonical outcome and fairness state \\
$\downarrow$ \\
Default and fairness reporting datasets \\
$\downarrow$ \\
Visualizer outputs, report manifests, and validation hubs
\end{tabular}
\end{center}

This is why the architecture uses object contracts and manifests instead of a one-off script. The same evidence chain can support quantum efficiency validation, clinical fairness validation, and future cross-domain comparisons while keeping domain-specific behavior in the appropriate child objects.

\section{System Architecture}

Figure~\ref{fig:architecture_draft2} presents the architecture as a technical summary of the workflow. The figure is intentionally high-level: it shows the direction of evidence flow, while Table~\ref{tab:stage_objects} gives the implementation mapping. At the top, problem formulation and data understanding define why the two applications belong in the same study. The control layer receives repository events or user entry paths, applies configuration and validation rules, and drives execution. The execution layer performs collection, cleaning, preparation, modeling, and evaluation. A governed adapter and sidecar layer then normalizes outcomes across quantum and clinical settings. The fairness--performance reporting layer combines inherited performance summaries with canonical fairness records, default/fairness master datasets, visualizer artifacts, report manifests, and validation hubs. Finally, the artifacts layer preserves the evidence produced by each stage.

In operational terms, the control layer is realized through configuration, orchestration, and validation entry points that correspond to the repository's local review behavior. The same logic that governs stage ordering, reproducibility, and merge validation is what allows the architecture to be inspected and rerun consistently by reviewers and collaborators.

\section{Detailed Description of Each Stage}

\subsection{Business \& Data Understanding}
The terms \textit{Business Understanding} and \textit{Data Understanding} are used here as research-methodology terms, not as narrow business-process labels. Business Understanding means identifying the real-world harm mechanism and the research objective. In this project, the harm mechanism is that partial-feedback policies route scarce resources using context whose quality is not equal across all groups or flows. The business/research objective is therefore to measure and mitigate repeated allocation harm while preserving utility.

Data Understanding means identifying the observable patterns that make that harm mechanism testable. In the clinical setting, those patterns include site, cohort, service line, test distribution, retesting capacity, escalation or diagnostic-routing action, latency, reward, success, and EQUITAS audit/mitigation fields. In the quantum setting, they include source testbed, scenario, allocator, route outcome, latency, success, reward, and evaluator state. The point of the data-understanding phase is to determine which fields explain the common structure between domains and which fields expose the paths that harm or mediate outcomes most strongly.

\subsection{Data Collection Stage}
The collection stage is built from \texttt{LocalBackupManager}, \texttt{GoogleDriveBackupManager}, the registry-based \textit{latest-state} lookup path, and the clinical fixture/evaluator builders listed in Table~\ref{tab:stage_objects}. \texttt{GoogleDriveBackupManager} resolves the shared-drive source and identifies the relevant storage root. \texttt{LocalBackupManager} provides the local-facing wrapper around that storage path. The registry logic then indexes available saved states and selects compatible artifacts. For clinical runs, the fixture builder creates controlled non-PHI clinical records. Together, these parts convert either the shared-drive namespace or the clinical fixture definition into concrete executable inputs for the pipeline.

\subsection{Data Cleaning Stage}
The cleaning stage is centered on \texttt{MultiRunEvaluator}, adapter normalization, and canonical outcome resolution. Retrieved quantum state is inspected for model identity and evaluator structure, while adapter-side resolvers convert raw state dictionaries into canonical outcome records. When evaluator summaries are incomplete or not normalized, summary-contract repair keeps downstream consumers stable. The output is a cleaned and contract-consistent evaluator or outcome state that downstream analysis can trust.

\subsection{Data Preparation and Exploration Stage}
This stage now includes both the inherited \texttt{state\_analysis.py} workflow and the generalized \texttt{StateReporting} object model. The inherited state-analysis code continues to support master datasets and validation for existing quantum outputs. The newer state-reporting layer adds an abstract \texttt{StateAnalysis} contract with \texttt{QuantumStateAnalysis} and \texttt{ClinicalStateAnalysis} children. \texttt{StateReporting} can emit default reporting datasets that preserve legacy performance methodology and fairness reporting datasets that include group, site, cohort, action, allocation, and EQUITAS fields. The result is a single reporting workflow that can produce comparable tables even when some fields are null in one domain.

\subsection{Modeling and Mining Stage}
The mining stage is implemented through two execution families. Legacy quantum experiments still use \texttt{QuantumExperimentRunner}, \texttt{MultiRunEvaluator}, and the allocator objects responsible for route qubit allocation. Clinical experiments now use \texttt{ClinicalExperimentEvaluator}, \texttt{ClinicalExperimentRunner}, \texttt{ClinicalEnvironment}, clinical allocator objects, and a clinical model registry containing \texttt{Oracle}, \texttt{ClinicalBaseline}, \texttt{FairAllocatorV1}, and \texttt{ClinicalBanditUCB}. This design keeps the old quantum runner intact while proving that the framework can plug in a non-quantum environment and compare model families through the same fairness/reporting surface.

\subsection{Evaluation Stage}
The evaluation stage is owned by evaluator and workflow objects. On the quantum side, \texttt{MultiRunEvaluator} computes winner counts, average gap, average efficiency, oracle reward, and scenario-level measures. On the clinical side, \texttt{ClinicalExperimentEvaluator} runs model decisions through the clinical environment and allocator/EQUITAS path, then exposes fairness states. \texttt{FairnessEvaluationService} and \texttt{FairnessWorkflow} normalize those states, build governed fairness summaries, and persist sidecar artifacts. The output is a reviewer-visible performance and fairness record that can be consumed directly or passed into reporting.

\subsection{Fairness--Performance Reporting Layer}
This layer now combines performance metrics, canonical fairness fields, model-stratified summaries, visualizer outputs, and report manifests. \texttt{FairnessByModelSummaryBuilder} records per-model parity gaps and group summaries. \texttt{EquitasFairnessMediator} provides an environment-agnostic audit/mitigation seam with clinical and quantum child mediators. \texttt{ExperimentVisualizer} defines the visualizer contract, while \texttt{ClinicalExperimentVisualizer} renders clinical model-efficiency and latency figures. \texttt{ReportManifestBuilder} ties fairness artifacts, default/fairness master datasets, visualizer outputs, and source lineage into \texttt{report\_manifest.json}. The output is a set of tradeoff tables, comparative plots, validation-hub inputs, and review-facing fairness-versus-performance summaries.

\subsection{Review Delivery and Replication Outputs}
The final stage uses notebooks, shared runners, visualizers, report manifests, validation hubs, and CI artifacts. The prior stages generate saved state, metrics, fairness summaries, master datasets, and plots. Notebook runners keep logic in reusable code while exposing a transparent interface for users. The clinical reproducibility notebook produces fairness artifacts, default/fairness master datasets, visualizer figures, and \texttt{report\_manifest.json}. The fairness validation hub consumes that manifest to verify preliminary fairness evidence. The output is a set of figures, tables, rerunnable notebook artifacts, and provenance records that support review and replication.

\section{Implemented Validation Gates}

The current system is guarded by two major smoke workflows. The Drive fairness smoke runs the legacy old-pipeline quantum wrapper over the default/no-testbed path plus the legacy quantum testbeds and saved-state paths from \quantumPaper. This gate ensures that the fairness sidecar does not break known legacy experiment paths. The clinical fairness smoke runs the embedded clinical evaluator with \texttt{Oracle}, \texttt{ClinicalBaseline}, \texttt{FairAllocatorV1}, and \texttt{ClinicalBanditUCB}, asserts sixteen fairness outcomes, checks model-stratified summaries, and runs focused tests for reporting, visualizers, notebooks, and validation hubs. These gates are path-scoped so expensive checks run when experiment, reporting, notebook, dependency, or workflow code changes, not on unrelated documentation-only edits.

\section{User Workflow}

\subsection{Setup and Installation}
Users can interact with the system through four supported paths. \texttt{Notebook path.} Open the quantum, clinical, or reporting notebook and run the shared-runner cells. \texttt{Local path.} Clone the repository, create a virtual environment, set \texttt{PYTHONPATH=Evaluation\_Framework}, and run the required scripts or notebooks. \texttt{GCP path.} Provision or enter a VM, run the startup script, and then use the batch or dynamic runner scripts for longer experiments. \texttt{CI path.} Push or open a pull request that touches experiment-impacting paths and let the scoped workflows validate the corresponding smoke surfaces.

\subsection{How the User Runs the System}
Once setup is complete, the user chooses a stage or full workflow. In the local path, that usually means running focused tests, the old-pipeline fairness smoke, or the clinical notebook pipeline. In the notebook path, it means configuring an experiment in a shared runner and executing the evaluator-backed workflow. In the GCP path, it means invoking the startup and batch scripts and allowing the run to complete remotely. These paths are different entry points into the same execution, fairness, reporting, and validation structure.

\subsection{How the User Interprets Results}
The user interprets results through shared artifact conventions. Quantum framework-state and model-state outputs remain under the shared data lake. Fairness sidecar outputs are written under \texttt{datalake/fairness/...}. Reporting outputs are written as default and fairness master datasets. Visualizer outputs generate figures and visualization manifests. \texttt{report\_manifest.json} ties those outputs back to the exact source artifacts that produced them. This means a figure, table, or validation claim can be traced to generated data rather than reconstructed manually.

\subsection{How the User Shares Results}
Because the data lake is shared-drive based and the reporting layer writes manifest-backed artifacts, completed runs can be made visible to collaborators without a separate export pipeline. Results written to \texttt{quantum\_data\_lake/}, \texttt{datalake/fairness/}, and \texttt{datalake/reports/} support team inspection, notebook-based reanalysis, and future validation-hub or portal interfaces.

\section{Architecture Overview}
%% ─────────────────────────────────────────────────────────────

This architecture writeup describes the implementation of the code review
repository used for the DSCI 601 assignment. The repository is organized as a
small, reproducible \textit{Hybrid CRISP+ML} pipeline that mirrors a real
machine-learning workflow while remaining fast enough for reviewers to install,
run, test, and inspect locally. The main design goal is to provide a
reviewer-safe system with clear stage boundaries, deterministic behavior, and
enough implementation detail that another party can recreate the workflow and
reproduce the same outputs.

The system is structured around seven explicit lifecycle stages: data
collection, data cleaning, data preparation, data exploration and
visualization, data mining, evaluation, and results postprocessing and
visualization. These stages were chosen so that the implementation follows a
recognized methodology while also satisfying the code review assignment's
requirement for separable runnable code paths.

\subsection{Configuration Layer}
The configuration layer defines the parameters controlling each run. In the
implementation repository, a typed configuration object is loaded from a single
JSON file. The configuration records the random seed, record counts,
train/test split, invalid-record frequency, missingness frequency, bias level,
fairness threshold, and output directory.

This layer separates experiment settings from stage logic. A reviewer can
change the run size, seed, or output location without modifying source code.
This design also supports reproducibility: the same configuration and seed
produce the same stage artifacts.

\subsection{Stage Layer}
Each lifecycle step is implemented as its own concrete class under a shared
\texttt{PipelineStage} interface. The concrete classes are:
\texttt{DataCollectionStage}, \texttt{DataCleaningStage},
\texttt{DataPreparationStage}, \texttt{DataExplorationStage},
\texttt{DataMiningStage}, \texttt{EvaluationStage}, and
\texttt{ResultsPostprocessingStage}. This keeps the implementation
object-oriented and gives every phase a narrow, testable responsibility.

The data collection stage generates a tiny synthetic dataset preserving
fairness-relevant structure such as group membership, label, and feature
availability. The cleaning stage validates records, removes intentionally
corrupted examples, and standardizes the schema. The preparation stage converts
clean records into train/test artifacts. The exploration stage writes small
summary tables. The data mining stage performs the lightweight modeling step.
The evaluation stage computes utility and fairness-oriented metrics. The final
postprocessing stage creates reviewer-facing tables and simple figures.

\subsection{Context and Orchestration Layer}
The orchestration layer connects stages without tightly coupling their
implementations. A shared pipeline context carries metadata, artifact paths, and
stage results across the run. The \texttt{ReviewPipeline} object builds the
ordered stage list and executes it either end-to-end or until a named stopping
point.

After execution, the orchestrator writes a manifest file containing the active
configuration, stage outputs, and stage summaries. This manifest is the
reproducibility record for a run: it identifies what was executed, where
artifacts were written, and what each stage reported.

\subsection{Interface Layer}
The user-facing interface is a command-line module with one command for each
required stage plus two orchestration commands: \texttt{run-all} and
\texttt{merge-check}. The command-line interface is part of the architecture
because it makes the repository reviewable as software rather than as a single
notebook. Reviewers can execute a single stage, rerun the full workflow, or
perform the same merge validation path used by the repository.

%% ─────────────────────────────────────────────────────────────
\section{Framework and Component Choices}
%% ─────────────────────────────────────────────────────────────

The implementation intentionally uses a minimal technical stack. The pipeline
is written with standard-library Python and a small package structure, with
\texttt{requirements.txt} as the reviewer-facing installation path. This choice
reduces environmental friction and avoids requiring a heavy machine-learning
stack for a code review assignment.

JSON is used for configuration, records, predictions, evaluation summaries, and
manifests because it is human-readable, easy to validate, and supported without
third-party libraries. CSV is used for summary tables because it is familiar to
reviewers and easy to inspect. SVG is used for lightweight charts because it
provides visual output without requiring plotting dependencies.

These choices prioritize reproducibility, transparency, and short setup time
over raw performance. That tradeoff is appropriate for this deliverable: the
goal is not to train a large model, but to demonstrate that the project has a
clean architecture, separable code paths, tests, and a reliable validation
workflow.

%% ─────────────────────────────────────────────────────────────
\section{Data Storage and Reproducibility}
%% ─────────────────────────────────────────────────────────────

All artifacts are written beneath a stable output root defined by the active
configuration. Within that root, each stage writes to its own numbered
directory. This output design makes the workflow easy to inspect: every phase
leaves visible artifacts, intermediate outputs can be reused, and reviewers can
verify stage behavior without rerunning the entire system.

The output layout answers three practical questions: what configuration
produced this run, what each stage wrote, and what final metrics or figures
were generated. The final manifest connects those answers into a single
provenance record. Because the repository uses deterministic configuration and
fixed seeds, another party can reproduce the same artifacts by running the same
commands.

The expected recreation path is straightforward: create a fresh Python
environment, install from \texttt{requirements.txt}, run the full pipeline with
\texttt{configs/fast\_review.json}, inspect the generated results directory,
and run \texttt{merge-check}. This reproduces the reviewer-safe workflow, its
stage artifacts, and its validation checks.

%% ─────────────────────────────────────────────────────────────
\section{Conclusion}
%% ─────────────────────────────────────────────────────────────

The architecture emphasizes simplicity, determinism, and inspectability. It
uses a recognized CRISP+ML lifecycle as the backbone, object-oriented stage
classes as the implementation structure, and explicit command-line entry points
as the reviewer interface. This makes the system appropriate for the DSCI 601
code review assignment while also providing a reusable foundation for larger
fairness-aware data science workflows.
